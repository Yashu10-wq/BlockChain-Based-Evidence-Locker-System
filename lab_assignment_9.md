# Lab Assignment 9: Risk Management and Mitigation Strategies

**Hypothetical Project:** Blockchain-Based Digital Evidence Locker System
**Project Description:** A secure, web-based application designed for law enforcement agencies to register, track, and manage digital and physical evidence using blockchain technology to ensure an immutable chain of custody.

---

## 1. Technical Risks

### Risk 1.1: Blockchain Integration Complexity
*   **Description:** Integrating a blockchain ledger with a traditional relational database (PostgreSQL) may cause synchronization issues, slow performance, or corrupted hashes due to timestamp mismatches.
*   **Impact:** High - If the blockchain ledger fails, the core value proposition of "immutability" is compromised.
*   **Mitigation Strategy:** 
    *   Implement robust unit testing for the hashing algorithms.
    *   Ensure exact millisecond timestamps are generated by the backend server and passed consistently to both the blockchain service and the database.
    *   Create automated database repair scripts to handle any development-phase synchronization failures.

### Risk 1.2: Tech Stack Compatibility Issues
*   **Description:** Utilizing a modern stack (Next.js, Node.js, Express, PostgreSQL) may lead to version conflicts or platform-specific deployment issues (e.g., Next.js Turbopack failing on specific Windows environments).
*   **Impact:** Medium - Can halt development or delay deployment.
*   **Mitigation Strategy:**
    *   Standardize the development environment using Docker containers.
    *   Maintain a strictly version-locked `package.json` and `package-lock.json`.
    *   Disable experimental features (like Turbopack) if native bindings are unsupported on team machines, falling back to stable Webpack builds.

---

## 2. Security Risks

### Risk 2.1: Unauthorized Access to Evidence
*   **Description:** Users with lower privilege levels (e.g., Custodians or regular Officers) might find loopholes to view the entire system's custody chain or access evidence they do not hold.
*   **Impact:** Critical - Violates strict law enforcement confidentiality requirements.
*   **Mitigation Strategy:**
    *   Implement strict Role-Based Access Control (RBAC) middleware on all backend API routes.
    *   Enforce conditional rendering on the frontend (e.g., hiding the "Custody Chain History" module for non-admins).
    *   Conduct thorough penetration testing and code reviews focusing on endpoint security.

### Risk 2.2: Database Tampering
*   **Description:** A malicious insider with database access might attempt to manually alter custody logs or delete evidence records to cover up mishandling.
*   **Impact:** Critical - Destroys the legal integrity of the evidence.
*   **Mitigation Strategy:**
    *   Implement PostgreSQL triggers (`prevent_update_delete`) that block any `UPDATE` or `DELETE` commands on the `custody_logs` table at the database level.
    *   Use blockchain hashing; any forced manipulation will break the cryptographic chain, which will be immediately caught by the system's Integrity Audit module.

---

## 3. Operational Risks

### Risk 3.1: QR Code Scanning Failures
*   **Description:** Officers may operate in environments with poor lighting, or physical QR code labels may get damaged, preventing successful custody transfers.
*   **Impact:** Medium - Slows down operational efficiency and transfer workflows.
*   **Mitigation Strategy:**
    *   Implement a high-quality, fault-tolerant QR scanning library (`html5-qrcode`).
    *   Provide a fallback manual entry option where officers can type in an alphanumeric Evidence ID and a one-time transfer PIN if the camera fails.

### Risk 3.2: System Out-Of-Memory (OOM) Crashes
*   **Description:** The application frontend may consume excessive memory during compilation or runtime (e.g., Next.js `Array buffer allocation failed`), causing the UI to crash.
*   **Impact:** High - Prevents users from accessing the system.
*   **Mitigation Strategy:**
    *   Disable heavy caching mechanisms during development if RAM is limited.
    *   Implement pagination and lazy loading for data-heavy pages (like the Dashboard and Audit Ledger) so the browser doesn't overload.

---

## 4. Project Management Risks

### Risk 4.1: Scope Creep
*   **Description:** Stakeholders continually request new features (e.g., adding facial recognition, mobile native apps) beyond the initial requirements, extending the deadline.
*   **Impact:** High - Leads to budget overruns and delayed launch.
*   **Mitigation Strategy:**
    *   Strictly adhere to an Agile methodology with clearly defined sprint goals.
    *   Maintain a "Phase 2 Backlog" for new feature requests, ensuring the Minimum Viable Product (MVP) is delivered first.

### Risk 4.2: Team Knowledge Silos
*   **Description:** Only one developer understands the complex blockchain hashing logic. If they become unavailable, the team cannot fix bugs in that module.
*   **Impact:** Medium - Creates a single point of failure in the development team.
*   **Mitigation Strategy:**
    *   Enforce mandatory code reviews and pair programming for critical components.
    *   Maintain comprehensive, up-to-date documentation (like `walkthrough.md` or API specs) for all complex system architectures.
